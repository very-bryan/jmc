# 보안 체크리스트 (Security Checklist)

모든 프로젝트 론칭 전에 확인해야 할 보안 항목.
JMC 진만추 프로젝트에서 실제로 발견하고 수정한 취약점을 기반으로 작성.

---

## 1. 인증 (Authentication)

- [ ] JWT 시크릿이 하드코딩되어 있지 않은가? (ENV 또는 credentials만 사용)
- [ ] JWT 시크릿이 없을 때 앱이 시작되지 않도록 설정했는가?
- [ ] JWT 만료 시간이 적절한가? (7일 권장, 24시간 이하는 UX 불편)
- [ ] 인증코드가 하드코딩되어 있지 않은가? (개발용이라도 프로덕션 배포 전 제거)
- [ ] 결제 토큰 검증이 실제로 동작하는가? (PG사 API 연동)
- [ ] 카카오/구글 등 소셜 로그인 정보를 클라이언트가 직접 설정할 수 없는가?
- [ ] 소셜 로그인 시 서버에서 토큰을 검증한 후에만 유저 정보를 설정하는가?

## 2. 인가 (Authorization)

- [ ] 모든 API 엔드포인트에서 `current_user` 스코핑을 하고 있는가?
- [ ] 유저 A가 유저 B의 리소스를 ID 변경으로 접근할 수 없는가? (IDOR 방지)
- [ ] 정지(suspended) 유저가 API를 사용할 수 없는가?
- [ ] 졸업/비활성화 유저가 API를 사용할 수 없는가?
- [ ] 차단된 유저 간 상호작용(관심/좋아요/메시지)이 차단되는가?
- [ ] 관리자 패널에 역할 기반 접근 제어(RBAC)가 적용되어 있는가?
- [ ] 관리자 인증과 유저 인증이 완전히 분리되어 있는가?

## 3. Rate Limiting

- [ ] 인증 API에 IP 기반 rate limit이 있는가? (분당 5회 권장)
- [ ] 일반 API에 IP 기반 rate limit이 있는가? (���당 60회 권장)
- [ ] 유저별(JWT) rate limit이 있는가? (관심/차단/신고 등 어뷰징 방지)
- [ ] 관리자 로그인에 rate limit이 있는가? (분당 3회 권장)
- [ ] 사전등록/회원가입 등 공개 API에 rate limit이 있��가?
- [ ] 게시글/메시지 등 생성 API에 일일 제한이 있는가?

## 4. 데이터 노출 방지

- [ ] API 응답에 내부 필드(is_seed_user, paid, role 등)가 포함되지 않는가?
- [ ] 에러 응답에 스택 트레이스, 파일 경로, DB 정보가 포함되지 않는가?
- [ ] S3/스토리지 내부 IP가 소스코드나 API 응답에 노출되지 않는가?
- [ ] 유저 검색 시 존재 여부를 구분할 수 있는 응답 차이가 없는가? (열거 방지)
- [ ] 차단된 유저도 동일한 에러 응답(404)을 반환하는가? (403 아닌 404)
- [ ] 전화번호가 응답에 불필요하게 포함되지 않는��?
- [ ] 로그에 전화번호, 비밀번호, 토큰이 평문으로 기록되지 않는가?
- [ ] filter_parameter_logging에 phone, token, password, base64가 포함되어 있는가?

## 5. 입력 검증

- [ ] 프로필 업데이트 API에서 verification_level, role 등 보안 필드가 permit에 포함되지 않는가?
- [ ] 메시지 타입이 허용 목록(text, image)으로 제한되는가?
- [ ] 이미지 URL이 내부 S3에서 온 것만 허용되는가? (외부 URL 주입 방지)
- [ ] 파일 업로드 폴더가 화이트리스트로 제한되는가? (path traversal 방지)
- [ ] 업로드 파일의 타입(매직 바이트), 크기(10MB), Content-Type이 검증되는가?
- [ ] Admin 검색에서 LIKE 와일드카드가 이스케이프되는가? (sanitize_sql_like)
- [ ] 회원가입 시 클라이언트가 보안 필드(is_seed, paid, verification_level)를 설정할 수 없는가?

## 6. 동시성 / 데이터 무결성

- [ ] 관심 수락 시 중복 Match 생성이 방지되는가? (lock + find_or_create_by)
- [ ] 초대코드 사용 시 동시 사용이 방지되는가? (with_lock)
- [ ] 초대코드 발급 시 중복 발급이 방지되는가? (with_lock)
- [ ] 관계(relationship) 생성 시 이미 활성 관계가 있는지 검사하는가?
- [ ] 관계 확인(confirm) 시 트랜잭션 + lock이 적용되는가?
- [ ] 관계 종료 시 양쪽 유저 상태가 올바르게 복귀하는가?
- [ ] 정지된 유저가 관계 종료로 active 복귀하지 않는가?
- [ ] 졸업한 유저가 관계 종료로 active 복귀하지 않는가?

## 7. CORS / 네트워크

- [ ] CORS origins가 와일드카드(*)가 아닌 실제 도메인만 허용하는가?
- [ ] 프로덕션에서 SSL이 강제되는가?
- [ ] DB 포트가 외부에 노출되지 않는가? (Docker 내부 네트워크만)
- [ ] DATABASE_URL이 ENV에만 있고 코드에 하드코딩되지 않는가?

## 8. 파일 업로드

- [ ] 파일 크기 제한이 있는가? (10MB 권장)
- [ ] 파일 타입을 매직 바이트로 검증하는가? (확장자만으로 불충분)
- [ ] Content-Type 허용 목록이 있는가? (image/jpeg, image/png, image/webp 등)
- [ ] 업로드 폴더가 화이트리스트로 제한되는가?
- [ ] 이미지 변환 명령어에 Shellwords.escape가 적용되는가? (커맨드 인젝션 방지)
- [ ] 변환 실패 시 원본 파일을 그대로 업로드하지 않는가?

## 9. WebSocket / 실시간

- [ ] WebSocket 구독 시 유저 권한을 검증하는가?
- [ ] 구독 후 conversation_id 변조로 다른 대화에 접근할 수 없는가?
- [ ] WebSocket 메시지 전송 시 참여자 재검증을 하는가?

## 10. Admin 패널

- [ ] 관리자 비밀번호 최소 길이(8자)가 설정되어 있는가?
- [ ] 기본 비밀번호(admin123)가 프로덕션에서 사용되지 않는가?
- [ ] 시드 파일 비밀번호가 ENV에서 읽어오는가?
- [ ] 세션 타임아웃이 설정되어 있는가? (30분 권장)
- [ ] 로그인 시 reset_session으로 세션 고정 공격을 방어하는가?
- [ ] 역할별 접근 제어(require_role!)가 모든 컨트롤러에 적용되어 있는가?
- [ ] 전화번호 검색이 super_admin만 가능한가?
- [ ] 관리자 로그인 rate limit이 있는가? (분당 3회)
- [ ] 주요 관리자 액션(유저 조회, 제재, 신고 처리)에 감사 로깅이 있는가?

## 11. 로깅 / 모니터링

- [ ] 인증 성공/실패가 로깅되는가?
- [ ] 로그에 전화번호가 마스킹되는가? (***5678)
- [ ] 관리자 액션이 감사 로깅되는가?
- [ ] 에러 로그에 내부 인프라 정보가 포함되지 않는가?

## 12. 의존성 / 공급망

- [ ] rack-attack (또는 동등한 rate limiter)이 설치되어 있는가?
- [ ] Gemfile.lock / package-lock.json이 git에 커밋되어 있는가?
- [ ] npm audit / bundle audit에 critical 취약점이 없는가?
- [ ] brakeman (또는 동등한 보안 스캐너)이 설치되어 있는가?
- [ ] brakeman 경고가 0건인가?

## 13. 개인정보 보호 (한국 PIPA)

- [ ] 개인정보 처리방침이 작성되어 앱/웹에 공개되어 있는가?
- [ ] 가입 시 수집/이��� 동의를 받고 있는가?
- [ ] 탈퇴/졸업 시 데이터 삭제 로직이 구현되어 있는가?
- [ ] 개인정보 보호책임자가 지정되어 있는가?

---

## 사용 방법

1. 프로젝트 론칭 전에 이 체크리스트를 처음부터 끝까지 확인
2. 각 항목에 체크하면서 해당 코드를 직접 확인
3. 미체크 항목은 수정하거나, 의도적으로 스킵하는 경우 사유 기록
4. 론칭 후에도 주기적으로(월 1회) 재확인

## 도구

- **Rails 보안 스캐너**: `bundle exec brakeman`
- **Node 취약점 검사**: `npm audit`
- **Ruby 취약점 검사**: `bundle audit check` (gem install bundler-audit)
- **gstack 보안 감사**: `/cso` (전체 자동 감사)
- **gstack 코드 품질**: `/health` (종합 점수)
